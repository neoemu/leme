# Contributing to Leme

Thanks for your interest! Leme is a native macOS Kubernetes IDE built with
SwiftUI. Issues and pull requests are welcome.

## Prerequisites

- macOS 26 (Tahoe) or later
- Xcode 26 or later, with the **Metal Toolchain** component
  (`xcodebuild -downloadComponent MetalToolchain`) — required by SwiftTerm's shaders
- [XcodeGen](https://github.com/yonaskolb/XcodeGen) (`brew install xcodegen`)
- `kubectl` (and `helm` for the Helm features) in your PATH
- A kubeconfig with at least one cluster to test against

## Building

```bash
git clone https://github.com/neoemu/leme.git
cd leme

# The Xcode project is generated — project.yml is the source of truth
xcodegen generate

open Leme.xcodeproj   # Build: Cmd+B, Run: Cmd+R
```

Or from the command line:

```bash
xcodebuild -project Leme.xcodeproj -scheme Leme -configuration Debug \
  -destination 'platform=macOS' -derivedDataPath build/DerivedData build
```

## Running tests

```bash
xcodebuild -project Leme.xcodeproj -scheme Leme -configuration Debug \
  -destination 'platform=macOS' -derivedDataPath build/DerivedData \
  test -only-testing:LemeTests
```

## Project conventions

- **Never edit `Leme.xcodeproj` directly** — it is generated. Change
  `project.yml` and run `xcodegen generate`. After adding/removing source
  files, regenerate and commit the updated project.
- **Swift 6 strict concurrency** is enabled. Services are actors, view models
  are `@MainActor @Observable`.
- UI changes to the sidebar or resource tables must follow the contract in
  [`docs/ui-consistency.md`](docs/ui-consistency.md).
- Destructive operations must respect the production guard-rails
  (type-to-confirm via `DangerConfirmationSheet` on clusters detected/marked
  as production).
- Pure logic (parsers, detection rules) belongs in testable static functions —
  see `ProblemsViewModel` and `HelmService` for the pattern, and add unit
  tests in `LemeTests`.
- The app icon is generated by `scripts/generate-icon.swift`
  (`swift scripts/generate-icon.swift <outdir>`).

## Pull requests

1. Fork and create a feature branch.
2. Make sure the build and unit tests pass locally (CI runs both on every PR).
3. Describe what you changed and how you tested it against a real cluster.
4. Keep PRs focused — one feature or fix per PR.

## Reporting bugs

Open an issue with: macOS and Leme versions, what you did, what you expected,
what happened, and (if relevant) the Kubernetes distribution/version. Never
paste kubeconfigs, tokens, or secret values into issues.
